# cb-qa — 커리어블룸 품질 검증

content-builder 산출물을 검증한다. 핵심 원리는 **"존재 확인"이 아니라 "교차 비교"** 다 — 정의된 규칙(브랜드 토큰·링크 목록·반응형 관례)과 실제 코드를 나란히 대조해 불일치를 찾는다.

## 점검 절차
1. **빌드 게이트**: `cd frontend && npm run build` — 실패 시 즉시 Blocker, 나머지 점검 전에 보고.
2. **결정적 점검 스크립트 실행** (누락 방지):
   - `bash .claude/skills/cb-qa/scripts/check-brand.sh` — 하드코딩 HEX, 비표준 폰트, target=_blank 누락 등 grep 기반 위반 탐지.
   - `bash .claude/skills/cb-qa/scripts/check-links.sh` — 코드 내 모든 외부 URL 추출 + 도달성 확인.
3. **수동 교차 비교** (스크립트로 못 잡는 의미적 검증): 아래 체크리스트.
4. **심각도 분류 + 리포트 작성**.

## 검증 축 (교차 비교 대상)

### 1. 브랜드 일관성
- 컬러: `#RRGGBB` 하드코딩이 className에 없는가? → 토큰(`sage`/`coral`/...)으로 대체 필요. **정당한 예외**(위반 아님): ① `index.css`의 CSS 변수 정의부, ② SVG 그라데이션 `stop-color`, ③ **외부 SNS 공식 브랜드 컬러**(카카오 `#FEE500`, 네이버 `#03C75A`, 인스타 `#833AB4/#FD1D1D/#FCAF45` 등) — 이 색들은 해당 서비스 정체성을 따라야 하므로 토큰화하면 안 된다.
- 폰트: 제목은 `font-serif`/`heading-display`, 본문은 `font-sans`인가? 임의 폰트 클래스 없는가?
- 그림자/라운드: sage 틴트 그림자, `rounded-2xl/3xl` 관례를 벗어난 이질적 스타일 없는가?

### 2. 반응형
- 신규 레이아웃에 `md:`/`lg:` 분기가 있는가? 데스크톱 전용으로 끝낸 곳 없는가?
- Hero/페이지 상단에 헤더 보정 패딩(`pt-24`+)이 있는가?
- 모바일 flex-column 버튼 정렬(`items-center`) 누락 없는가?

### 3. 인터랙션 무결성
- 절대배치 장식 컨테이너에 `pointer-events-none`이 있는가? (없으면 버튼 클릭 차단 → Blocker)
- 클릭 요소(`Link`/`a`/`button`)가 장식 레이어에 덮이지 않는가?

### 4. 링크 유효성 (경계면 교차 비교)
- **단일 출처**: `frontend/CLAUDE.md`의 "외부 링크" 목록.
- 코드 내 실제 `href`/`to`를 이 목록과 대조. 오타·구버전 URL·누락 링크 탐지.
- 외부 링크에 `target="_blank" rel="noopener noreferrer"` 세트 존재 여부.
- 내부 라우트(`to="/about"` 등)가 `App.jsx` 라우터에 실제 존재하는가?

### 5. 접근성 (기본선)
- `<img>`에 의미있는 `alt`가 있는가?
- 아이콘 버튼에 `aria-label`이 있는가? (예: 모바일 메뉴 버튼)
- 텍스트/배경 대비가 극단적으로 낮지 않은가(warm-gray on cream 등 경계 케이스 플래그).

## 심각도 분류
| 심각도 | 기준 | 예 |
|--------|------|----|
| **Blocker** | 배포 불가 | 빌드 실패, 깨진 링크/라우트, pointer-events로 클릭 막힘 |
| **Major** | 사용자 체감 결함 | 모바일 레이아웃 깨짐, 브랜드 토큰 위반, alt/aria 누락 |
| **Minor** | 다듬기 | 미세 간격, 카피 오타, 일관성 미흡 |

## 리포트 형식
각 항목은 `파일:라인 — [심각도] 규칙 위반 — 수정 제안` 형태. 막연한 표현 금지.
최종에 **PASS / FAIL** 판정을 명시한다(Blocker 0개 + Major 합의 해소 시 PASS). 팀 모드에서는 `_workspace/A_qa_report.md`에 누적 기록하고, regression(이전 지적 재발) 여부를 먼저 확인한다.

## 점진적 QA
전체 완성 후 1회가 아니라, content-builder가 모듈을 완료할 때마다 해당 변경분을 즉시 검증한다. 조기 검증이 후반 누적 결함을 막는다.
